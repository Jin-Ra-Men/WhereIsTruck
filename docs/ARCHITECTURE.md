# WhereIsTruck 시스템 아키텍처

WhereIsTruck(어디야트럭)의 시스템 구성, 데이터 흐름, 핵심 로직을 정리한 문서입니다.

---

## 1. 서비스 개요

- **목적:** 실시간 위치 기반으로 노점·푸드트럭 정보를 공유하고, 사용자와 사장님을 신뢰 기반으로 연결
- **가치:** 실시간성, 신뢰 기반 검증, 지역 커뮤니티 활성화

---

## 2. 전체 아키텍처 개요

```
┌─────────────────────────────────────────────────────────────────┐
│                        클라이언트 (모바일)                         │
│                Flutter + Map API + Push (FCM)                     │
└───────────────────────────┬─────────────────────────────────────┘
                            │ HTTPS / WSS
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                         백엔드 API                                │
│                 NestJS (REST + Socket.io)                         │
│         · 인증(Firebase)  · 트럭/위치/사용자  · 실시간 이벤트      │
└───────────────┬─────────────────────────────┬───────────────────┘
                │                             │
                ▼                             ▼
┌───────────────────────────┐   ┌───────────────────────────────┐
│   PostgreSQL + PostGIS    │   │  Firebase (Auth, Push 등)      │
│   · 트럭/위치/사용자/찜    │   │  · 사장님/사용자 인증           │
│   · 공간 쿼리(반경 검색)   │   │  · 푸시 알림 (단골 트럭 영업 등) │
└───────────────────────────┘   └───────────────────────────────┘
                │
                │ (선택) 지도 타일/경로
                ▼
┌───────────────────────────┐
│  Map API (Google / Kakao) │
└───────────────────────────┘
```

- **클라이언트:** 모바일 앱에서 지도 표시, 주변 트럭 조회, 찜/알림, 사장님은 영업 시작·재고 공지
- **백엔드:** REST로 CRUD·인증, WebSocket(Socket.io 등)으로 실시간 위치·상태 전달
- **DB:** PostgreSQL + PostGIS로 위경도 저장 및 반경 n km 검색
- **외부:** Firebase(인증·푸시), Map API(지도·좌표)

---

## 3. 핵심 도메인 로직

### 3.1 위치 서비스 (내 주변 트럭 찾기)

- **입력:** 사용자 위경도 `(lat, lng)`, 반경 `r_km` (예: 2 km)
- **처리:** PostGIS 공간 쿼리로 “현재 영업 중”인 트럭 중 반경 내만 필터링
  - 예: `ST_DWithin(geom::geography, ST_MakePoint(lng, lat)::geography, r_km * 1000)`
- **출력:** 거리순/인기순 정렬된 트럭 목록 (이름, 위치, 메뉴 요약, 영업 상태 등)
- **실시간:** 사장님이 “영업 시작”하면 위치가 DB에 반영되고, Socket으로 주변 클라이언트에 브로드캐스트해 지도가 즉시 갱신되도록 설계

### 3.2 인증 시스템 (신뢰 기반 검증)

- **일반 사용자:** Firebase Auth 등으로 로그인, 프로필·찜 목록 저장
- **사장님 권한:**
  - 현장 사진 업로드 + EXIF 등 메타데이터 검증으로 “실제 노점” 여부 보조
  - 사용자 추천(리뷰/찜) 기반 집단지성으로 신뢰도 누적
- **백엔드:** Firebase Admin SDK로 토큰 검증 후, 사장님 전용 API(영업 시작/종료, 재고 공지 등) 접근 허용

### 3.3 실시간 (Socket.io / WebSocket)

- **이벤트 예시:**
  - 사장님: `truck:open`, `truck:close`, `truck:location`, `truck:stock_update`
  - 서버 → 클라이언트: 특정 지역/반경 내 트럭 목록 갱신, 단골 트럭 영업 시작 알림
- **위치 업데이트:** 주기적 또는 변경 시에만 전송해 트래픽 절감

### 3.4 비즈니스 모델 (참고)

- 프리미엄 노출: 일부 사장님에게 지도 상 상단/강조 노출 권한
- 타겟팅 푸시: “찜한 트럭이 영업 시작” 등 이벤트 기반 푸시 (Firebase Cloud Messaging 등)

---

## 4. 데이터 모델 개념 (핵심 엔티티)

- **User** — 일반 사용자 또는 사장님, Firebase UID, 프로필, 역할(owner/user)
- **Truck** — 노점/트럭 정보, 사장님 FK, 메뉴·사진·영업 상태(open/closed), 현재 위치(위경도)
- **Location** — 트럭의 위치 이력 또는 “현재 위치” (PostGIS geometry/geography)
- **Favorite** — 사용자–트럭 찜 관계 (단골 트럭 알림에 사용)
- **Recommendation / Review** — 사장님에 대한 사용자 추천·리뷰 (신뢰도 계산에 사용)

위치는 PostGIS의 `geometry(Point, 4326)` 또는 `geography` 타입으로 저장하고, 반경 검색 시 인덱스를 사용하는 것을 권장합니다.

---

## 5. 보안 및 환경

- **API 키·비밀:** Map API, DB URL, Firebase Admin 키 등은 환경 변수(`.env`)로 관리, 저장소에는 `.env.example`만 커밋
- **인증:** 모든 사장님 전용·사용자 개인 API는 토큰 검증 후 접근
- **위치 개인정보:** 위치는 “현재 세션/요청”에만 사용하고, 불필요한 장기 저장은 최소화

---

## 6. 인프라 (배포 시)

- **Backend:** AWS(GCP) VM/컨테이너에서 NestJS 실행
- **DB:** Managed PostgreSQL + PostGIS (RDS, Cloud SQL 등)
- **실시간:** 동일 서버에서 Socket.io 서버 실행 또는 별도 스케일 아웃
- **Push:** Firebase Cloud Messaging (또는 동일 계정의 FCM)
- **Map:** Google Maps API 또는 Kakao Map API (클라이언트/서버 용도에 맞게 키 분리)
